# lendsqr WalletService

## Overview
lendsqr Demo WalletService is a robust API for managing user wallets, allowing users to create wallets, perform transactions, and check balances securely.

## Features
- User authentication
- Wallet creation and management
- Deposit, withdrawal, and transfer transactions
- Transaction history tracking
- Secure JWT-based authentication

## Tech Stack
- **Backend:** Node.js, Express, TypeScript
- **ORM:** KnexJS
- **Database:** MySQL
- **Authentication:** JWT
- **Test:** Jest

## Installation

```sh
# Clone the repository
git clone https://github.com/Elijah57/lendsqr.git

# Navigate to the project folder
cd lendsqr

# Install dependencies
npm install

# Set up environment variables
cp .env.example .env

# Start the server
npm run dev
```


# üè¶ Wallet Service Documentation

## üìñ Overview
The **Wallet Service** manages user transactions, balances, and financial operations within our application. This document outlines the **ER Diagram** and core functionalities of the wallet service.

## üìå Entity-Relationship Diagram (ERD)
The **ER Diagram** visually represents the relationships between key database entities in the wallet system.

![ER Diagram](erd.png)  


### **üóÇÔ∏è Entities & Relationships**
1. **Users** üë§  
   - `id` (PK) ‚Äì Unique identifier for each user.  
   - `firstname` ‚Äì User's first name.  
   - `lastname` ‚Äì User's last name.  
   - `phoneno` ‚Äì User's phone number.  
   - `email` ‚Äì Unique email address.  
   - `password` ‚Äì Hashed password for authentication.  
   - `created_at` ‚Äì Timestamp when the user was created.  
   - `updated_at` ‚Äì Timestamp when the user was updated.  

2. **Wallets** üí∞  
   - `id` (PK) ‚Äì Unique identifier for each wallet.  
   - `user_id` (FK) ‚Äì References the user owning the wallet.  
   - `account_balance` ‚Äì Current wallet balance.  
   - `account_number` ‚Äì Unique identifies for each wallet, to facilitate transactions.  
   - `account_name` ‚Äì Full name of the user owning the wallet.  
   - `currency` ‚Äì Type of currency (defaults to NGN).  
   - `provider` ‚Äì Provider of the wallet service account (defaults to lendsqr-demo).  
   - `balance_last_updated` ‚Äì Timestamp when the wallet balance was updated.  
   - `type` ‚Äì Wallet type (defaults to `default`).  
   - `created_at` ‚Äì Timestamp when the wallet was created.  

3. **Transactions** üîÑ  
   - `id` (PK) ‚Äì Unique transaction ID.  
   - `sender` ‚Äì References the wallet {`account_name`, `account_number`}involved.  
   - `receiver` ‚Äì References the wallet {`account_name`, `account_number`}involved.  
   - `type` ‚Äì Transaction type (`deposit`, `withdrawal`, `transfer`).  
   - `amount` ‚Äì Transaction amount.  
   - `status` ‚Äì Transaction status (`pending`, `completed`, `failed`).  
   - `reference` ‚Äì Unique transaction reference code.  
   - `created_at` ‚Äì Timestamp of the transaction.  

4. **Banks** üí≥  - Banks in nigeria and their information

 	- `id` (PK) ‚Äì Unique record identifier.  
	- `name` ‚Äì Name of the entity.  
	- `code` ‚Äì Unique short identifier.  
	- `slug` ‚Äì URL-friendly unique identifier.  
	- `longcode` ‚Äì Optional long version of the `code`.  
	- `country` ‚Äì Country of the entity (default: `Nigeria`).  
	- `currency` ‚Äì Currency used (default: `NGN`).  
	- `created_at` ‚Äì Timestamp when the record was created.  
	- `updated_at` ‚Äì Timestamp when the record was last updated.  


## API Endpoints

### Authentication

#### 1. User Registration
**Endpoint:** `POST /api/v1/auth/signup` 

**Description:** This endpoint facilitate users account creation, on signup, user account is created and a wallet is automatically created for each user
**Request Body:**
```json
{
  "firstname": "John",
  "lastname": "Doe",
  "email": "johndoe@example.com",
  "phoneno": "+2345683954895",
  "password": "securepassword"
}
```
**Response:**
```json
{
	"status": true,
  "message": "User registered successfully",
	"userDetails": {
		"user": {
			"id": "426f4ee7-030e-11f0-94a9-0242ac150002",
			"firstname": "Chris",
```

# **üìö Libraries Used & Their Purpose**  

This document provides an overview of the dependencies used in this project, explaining their roles and why they were chosen.  

## **üîê Security & Authentication**  

- **`argon2`** (`argon2id`) ‚Äì  The OWASP (Open Web Application Security Project) Recommends using `argon2id` as best practices for securely hashing passwords using the Argon2 algorithm. It is Used for **password hashing**. Argon2 is a modern, secure, and memory-hard hashing algorithm that protects passwords against brute-force attacks.  

- **`jsonwebtoken`** (`jwt`) ‚Äì Used for **authentication** via JWT (JSON Web Tokens). It enables secure token-based authentication for user sessions.  

## **üåê Server & API Handling**  

- **`express`**  ‚Äì A lightweight and powerful web framework for Node.js. It simplifies API development by handling routes, middleware, and request-response logic.  

- **`cors`** ‚Äì Middleware that enables **Cross-Origin Resource Sharing (CORS)**, allowing the server to handle requests from different origins (e.g., frontend apps hosted separately).  Cors options for this project can be found in `src/configs`. At the momnet, it accepts request from all origins `*`, but could be configure.

- **`node-fetch`** ‚Äì A lightweight module for making **HTTP requests** from the server. It is used in this project to fetch external APIs (paystack, adjutor).  


## **üóÑÔ∏è Database & Query Handling**  

- **`knex`**  ‚Äì A SQL query builder that provides a structured way to interact with databases, supporting **migrations, transactions, and connection pooling**.  The configurations can be found in the `knexfile.ts`
- **`mysql2`**  ‚Äì A high-performance MySQL driver for Node.js, used by **Knex.js** to connect and execute queries efficiently.  

## **üîß Environment Configuration**  

- **`dotenv`** (`^16.4.7`) ‚Äì Loads **environment variables** from a `.env` file, keeping sensitive data like database credentials and API keys secure.  

## **üõ¢Ô∏è Database Setup**  

- **Database Engine:**  
  - The project uses **MySQL 8.0 (Oracle)** as the database engine, running inside a **Docker container**.  
  - **Image:** `mysql:8.0-oracle`  
  - **Storage:** A **persistent volume** is used to ensure data is not lost when the container stops or restarts.  

- **Adminer UI:**  
  - **Adminer** is included in the Docker setup as a lightweight web-based database management tool.  
  - It allows developers to interact with the database via a **simple web interface** instead of running raw SQL queries manually.

## **üõ†Ô∏è Architectural Design & Folder Structure**  


This project follows a **modular and OOP-based architecture**, ensuring **separation of concerns**, maintainability, and ease of collaboration.
---

## **üèóÔ∏è Architectural Design Overview**  
The backend is designed using the **MVC (Model-View-Controller) pattern**, with additional layers for **services, middlewares, utilities, and database interactions**. This ensures:  

‚úîÔ∏è **Separation of concerns** (each module has a single responsibility).  
‚úîÔ∏è **Scalability** (new features can be added without breaking existing logic).  
‚úîÔ∏è **Reusability** (common logic is extracted into reusable services and utilities).  

The system consists of the following core components:  

1. **Controllers** ‚Äì Handle incoming requests and responses.  
1. **APIs** ‚Äì Handle request and responses from API services.  
2. **Services** ‚Äì Contain business logic and communicate with repositories.  
3. **Middlewares** ‚Äì Handle request validation, authentication.  
4. **Utilities (Utils)** ‚Äì Reusable helper functions (e.g., hashing, JWT handling, generating Transaction Reference).  
5. **Database (DB Layer)** ‚Äì Manages migrations, schemas, repositories, and connections.  
6. **Configuration (Configs)** ‚Äì Centralized application settings (e.g., environment variables, database config).
